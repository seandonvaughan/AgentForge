## Risk Assessment

### 🟡 Code Changes: `proposal-to-backlog.ts`

**What changed:**
1. **Lines 67-68:** Comment text rewritten to avoid literal `TODO(autonomous): ...` pattern that triggered false positives when the scanner ran against its own source file.
2. **Lines 252-259:** `sanitizeItems()` extended to sanitize `description` field (previously only `id` and `title`), using the same `strip()` helper.

**Why it matters:**  
The `sanitizeItems()` call is an **output boundary guard** — it's the last filter before backlog items reach the backlog schema and flow into the `backlog-scorer` agent. Any uncleaned HTML/Markdown comment closers (`-->`, `*/`) in description fields can corrupt scorer prompts and skew decision logic.

**Confidence in change:** ✅ HIGH
- Both changes are **pure data cleaning** (comments rewritten, sanitization extended).
- No business logic modifications.
- No kill-switch, cost-tracking, or git-ops paths affected.
- All 10 existing unit tests pass.

## Missing Test Coverage (Critical Gaps)

### ⚠️ **Gap 1: Description field not validated in sanitization test**

**Location:** `tests/autonomous/unit/proposal-to-backlog.test.ts`, test `"strips --> artifacts from non-todo-marker items at the output boundary"` (lines 191–212)

**Issue:**  
The test creates backlog items with `-->` in their description fields (session errors, task outcomes) but **does not assert** that descriptions are actually stripped:

```typescript
// Test data has:
{ id: 's1', agent: 'coder', error: 'TypeError: undefined -->', ... }
{ taskId: 't1', description: 'broken task -->', ... }

// But assertions only check title and id:
expect(items.every(i => !i.title.trimEnd().endsWith('-->'))).toBe(true);
expect(items.every(i => !i.id.trimEnd().endsWith('-->'))).toBe(true);
// Missing: expect(items.every(i => !i.description.trimEnd().endsWith('-->'))).toBe(true);
```

**Impact:** If future code refactors inadvertently break description sanitization, **this test will not catch it**. Corrupted descriptions would flow to the scorer agent undetected.

**Recommended test:**  
```
sanitization-test: description-field-strip-html-closers
  Input: { id: 's1', error: 'TypeError: undefined -->', confidence: 0.9 }
  Expected: description should NOT end with '-->'
  Example: "Session s1 failed with: TypeError: undefined" (no closer)
```

### ⚠️ **Gap 3: Block comment closers (`*/`) not tested for stripping**

**Issue:**  
The code strips both `-->` (HTML) and `*/` (block comment closers):

```typescript
const strip = (s: string): string =>
  s.replace(/\s*-->\s*$/, '').replace(/\s*\*\/\s*$/, '').trim();
```

But existing tests only validate `-->` stripping. If `*/` stripping breaks, tests won't catch it.

**Recommended test:**  
```
sanitization-test: description-strip-block-comment-closer
  Input: description: "Task failed with parse error */"
  Expected: "Task failed with parse error" (no closer)
```

---

### ⚠️ **Gap 4: No edge case for multiple/nested closers**

**Issue:**  
What if text contains both `-->` and `*/`? Or multiple closers in sequence? The code applies `.replace()` twice, so it should work, but no test validates this.

**Recommended test:**  
```
sanitization-test: multiple-closers
  Input: description: "Error: undefined --> and also */ here"
  Expected: "Error: undefined and also here" (both stripped)
```

### ⚠️ **Gap 5: No integration test validating end-to-end backlog correctness**

Unit tests validate individual components (comment scanning, sanitization, deduplication) but don't validate the full pipeline in a realistic scenario:

```
integration-test: proposal-to-backlog-end-to-end

```
