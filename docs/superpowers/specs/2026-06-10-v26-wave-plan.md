**Date:** 2026-06-10 · **Status:** PROPOSED (operator review) · **Provenance:** strategic planning pass grounded against main at `fe40bd1` (post-epic #308), the objective-mode redesign spec (`docs/superpowers/specs/2026-06-06-objective-mode-redesign.md` — currently untracked, lives only in the operator's working tree; commit it alongside this plan), the v24 design (`docs/superpowers/specs/2026-06-09-v24-platform-upgrades-design.md`), the live cycle artifacts of epic `5242ca92`, `.agentforge/memory/cycle-ledger.jsonl`, and the in-flight v25 PRs #309–#314 + F1/F2.

**Locked operator decisions honored here:** (1) the legacy crumb pipeline is DELETED now that the $300 home-repo epic passed; (2) the next external project is **AgentForge Cloud** (new local-first repo, device registration for plans and/or API keys, recommended foundation: Anthropic Managed Agents) — AgentForge itself must be solid first; (3) Claude-first one-product, codex auxiliary.

## 1. Thesis

v25 made the product *honest* (provider-accurate agents, Claude-first forge, a team that matches reality); v26 makes it *trustworthy*. The $300 home-repo epic proved the objective-mode architecture end to end — 19 of 21 children completed (`.agentforge/cycles/5242ca92-*/checkpoint-execute.json`), one coherent merged PR (#308) — but the same run exposed exactly where trust still leaks: the epic-level VERIFY tests the operator's tree instead of the integration worktree, a failed resume corrupts the phase checkpoint and clobbers the spend artifacts that feed calibration (the live `cycle-ledger.jsonl` entry for the epic records $3.93 of overhead and **zero** execution spend for a run that completed 19 children — the calibration store's primary input was destroyed by its own machinery), VERIFY worker sizing collapses under memory pressure because it reads `freemem()`, and the planner still declares wrong registration files because nothing teaches it the repo's wiring conventions. v26 therefore runs five tightly-scoped autonomous epics plus one supervised deletion: fix verification and resume durability first (W1), then kill the legacy crumb pipeline under human supervision (W2) — deleting the incentive structure the 2026-06-06 forensics condemned — then ground the planner in repo conventions (W3), close the operator-facing polish gaps (W4), deepen the Claude Code session surface (W5), and mature calibration on top of now-durable artifacts (W6). When W1+W2 are merged, AgentForge has ONE pipeline, verified where the code actually lives, with spend data that survives failure — that is the bar for "solid", and it is the gate for kicking off AgentForge Cloud (§6).

### W1 — Verification & resume integrity ($100, autonomous)

The keystone. Every later wave runs as an epic and inherits whatever VERIFY and resume do; fix them first.

**Budget justification:** 6 opus-tier children (VERIFY relocation, worker heuristic, checkpoint guard, resume cost restore, per-item cost durability, ledger idempotency) at $5–12 ≈ $48 + 3 sonnet children (tests/docs/event surface) at $2–5 ≈ $10. Est ≈ $58 + $2 decompose + $0.5 review. Band check: $100 → spendable $78.3, band $54.8–78.3 — estimate sits in band.

**Acceptance criteria:**
- A fixture epic cycle's STAGE 4 VERIFY demonstrably executes inside the integration worktree (assert on the recorded cwd in `tests.json`/events), with worktree dependencies provisioned first (reusing the per-child provisioning pattern from #279/#280).
- `run-verify-tests.mjs` worker count never drops below 2; under a simulated low-`freemem` reading it logs the heuristic decision and uses the available-memory estimate (or the env override) instead of collapsing.
- A resume that *fails* leaves `checkpoint-cycle.json` byte-identical to its pre-resume state (unit test with injected failure); a resume that succeeds preserves prior `completedPhases`.
- After kill + resume of a fixture epic, `cycle.json` `cost.totalUsd` ≥ the sum of per-item costs recorded in the extended execute checkpoint, and the final `spend-report.json` lists every completed child with non-zero actuals.
- `cycle-ledger.jsonl` ends with exactly one entry per cycleId reflecting the final (best) state after a fail-then-resume sequence.

**Risk notes:** touches `cycle-runner.ts` and execute-phase territory that F1 also touches — **must not launch until F1 and #309–#314 are merged** (single-flight rule). Relocating VERIFY into the worktree removes the accidental "does main still pass?" check; that is acceptable because the branch's own CI covers it and the operator-tree run was testing the WRONG code anyway. Memory heuristics on darwin are inherently fuzzy — the floor + override + logging is the safety net, not the estimate.

**Objective text (`/tmp/v26-w1-objective.txt`):**

> Make epic verification and resume durable and correct, end to end.
>
> Core (packages/core):
> - STAGE 4 VERIFY for objective cycles must run the full test suite INSIDE the epic integration worktree, not the operator's project root: thread the integration worktree path from the execute phase result into the VERIFY stage in packages/core/src/autonomous/cycle-runner.ts (STAGE 4, currently calling this.options.testRunner.run against the project root), give RealTestRunner (packages/core/src/autonomous/exec/real-test-runner.ts) an explicit cwd, and provision the worktree's dependencies first by reusing the same provisioning used by the per-child verify (packages/core/src/autonomous/phase-handlers/child-verify.ts). ## 5. What must NOT be an autonomous cycle — and why

**W2 (legacy deletion) is a supervised Claude Code session, not an epic.** The call, justified:

1. **The blast radius is cross-cutting, not decomposable.** The kill list's modules are imported by the modules that survive (`scoring-pipeline` ← `runtime-adapter`/`preview-cycle`/`cycle-runner`; `proposal-to-backlog` ← `preview-cycle`; root phase-handlers ← `self-eval/recorder.ts`, memory paths). The hard work is deciding the *boundary* — which exports get extracted before deletion — and that decision spans children. The epic planner decomposes by file surface; a deletion's correctness is global.
2. **Direct evidence the planner misjudges wiring:** child-4 of the operator-console epic declared the wrong registration file on a *constructive* task. A deletion task gives wrong-boundary errors a much worse failure mode: a child that deletes an import the next wave's child needed produces cascading red that burns budget on repair, and the gate's iron-law checks (declared files touched, tests present) are the wrong shape for "the value is what is no longer there."
3. **The deletion changes the machinery that would be executing it.** An epic deleting `cycle-runner.ts` branches while `cycle-runner.ts` orchestrates that epic is self-modification under load — the resume/checkpoint code paths being deleted are the same ones a mid-run failure would need.
4. **Cost is not the bottleneck; judgment is.** A supervised session on the Max plan costs effectively zero marginal dollars and a day of attention. The repo's own history (PR #43, #57: large single-session changes shipped in a day) shows attended sessions excel at exactly this shape.

**Also not autonomous:**
- **The AgentForge Cloud repo bootstrap** (§6): a fresh repo has no conventions, no calibration data, no CI to act as backstop. The first objective there runs *attended* with preview, per the scaffold-acceptance protocol.
- **Team/agent YAML and CLAUDE.md edits** (`.agentforge/agents/`, `team.yaml`, `CLAUDE.md`): owned by the operator + F2; cycles must never mutate them (the auto-reforge lesson, already enforced).
- **PR hygiene** (§8 pre-flight): five `gh` commands; automation would cost more than it saves at current volume (2 stale drafts + 1 stale cycle PR + 3 dependabot).

Everything else in v26 is deliberately autonomous — each wave is constructive, file-scoped, consumer-named, and preview-gated, which is the shape the epic machinery has now twice proven it handles.

### 6.1 Repo scaffold shape

### 6.2 Managed Agents architecture sketch (mapping to AgentForge concepts)

| Managed Agents primitive | AgentForge concept it becomes |
|---|---|
| **Sessions** (server-managed, per-session containers) | Cloud-side epic children: each child = one managed session in its own container, replacing local WorktreePool isolation when execution runs in-cloud. Local execution remains the default; cloud sessions are the BYO-nothing tier. |
| **Outcomes** (`define_outcome` rubric → iterate/grade/revise loop) | The epic review + fix-up loop as a managed service: a child's acceptance criteria become the outcome rubric; the platform's grade/revise loop replaces our per-child retry prompt for cloud-executed children. |
| **Vaults** (credential custody) | Device registration's credential model: a registered device deposits either an API key or plan-OAuth grant into a vault; sessions draw credentials from the vault — keys never live in cycle artifacts or prompts. |
| **GitHub repo resources** (clone/push via credential-injecting proxy) | The release phase for cloud children: clone/push without a local `GH_TOKEN`, replacing `exec/git-ops.ts` + `pr-opener.ts` mechanics for the cloud path. |
| **Multiagent coordinator** (≤20 agents / 25 threads) | Wave execution for cloud epics — maps cleanly onto the ≤20-child epics we already plan (the operator-console epic was exactly 20 children). |
| **Memory stores** | Cloud home for `.agentforge/memory/` + cost-priors so calibration follows the account across devices. |
| **Webhooks** | Cycle lifecycle events → console push (replaces SSE polling across the WAN). |

**Architecture stance:** local AgentForge keeps the self-hosted execution engine; Cloud is a thin orchestration/billing/registration layer that can *delegate* execution either to a registered device (local-first path) or to Managed Agents sessions (cloud path). The `packages/managed/` seam keeps Managed Agents (beta) churn contained.

### 6.3 What AgentForge itself must expose for Cloud (feeds back into v26/v27 backlog)

1. **API-key execution parity:** the `anthropic-sdk` transport must run the full agentic child loop headlessly (today's proven path is `claude-code-compat`, which assumes a logged-in CLI). Acceptance: one epic child end-to-end with `AGENTFORGE_RUNTIME=sdk` and only `ANTHROPIC_API_KEY` set.
2. **Prompt caching on the SDK path:** `cache_control` breakpoints on the stable prefix (digest, epic plan, per-agent system prompt) — spec §15.5 of the objective-mode redesign; the forensics found zero caching. Directly determines Cloud's unit economics.
3. **Headless auth:** `agentforge auth` — API-key storage in the OS keychain + an OAuth device-code flow for plan-based auth; the daemon reuses it for device registration.
4. **Remote-control surface stability:** the v5 REST + SSE API is already multi-instance parameterized (PR #277); Cloud needs it versioned and CORS/auth-hardened (the existing auth plugin + rate limiting are the base).
5. **Spend export:** the (W1-durable, W6-tiered) ledger and priors become the billing substrate — per-device, per-cycle usage rollups exposed via one endpoint.

Items 1–3 should be drafted as the first v27 wave in this repo once W1/W2 land; none block W3–W6.
