# AgentForge v3.1 Spec-to-Code Gap Matrix

**Prepared:** March 25, 2026  
**Audience:** AgentForge development team  
**Purpose:** Translate the current repository review into a v3.1 delivery matrix that maps documented product claims to actual code status and concrete implementation moves.

## Scope

This document is based on a repository-grounded review of:

- CLI entrypoints and user flows under `src/cli/`
- Forge, Genesis, builder, domain, scanner, and collaboration layers under `src/`
- Runtime orchestration, budget, feedback, reforge, and integration layers under `src/`
- Templates under `templates/`
- Tests under `tests/`
- Public docs and planning artifacts under `README.md`, `docs/`, and `commands/`

## Repository Snapshot

As of March 25, 2026, the repository currently contains:

- 10 domain directories under `templates/domains/`
- 59 domain agent YAMLs under `templates/domains/*/agents/`
- 97 total YAML files under `templates/`
- 93 TypeScript source files under `src/`
- 56 test files under `tests/`

## Review Constraints

- The analysis is code-grounded.
- The full test suite was **not executed** in this workspace because dependencies are not installed and `npm test` currently fails without `node_modules`.
- Where docs and code disagree, this document treats code as the current source of truth.

---

## Executive Summary

AgentForge's most complete and product-shaped subsystem is the static team-generation pipeline behind `agentforge forge`. The repository also contains a broader Genesis layer and a much richer v3 runtime/orchestration layer, but those systems are only partly integrated into the normal CLI experience.

The main pattern across the repo is:

1. The architecture is ahead of the product surface.
2. The core subsystems exist, but several are still scaffolds or local-only heuristics.
3. Documentation and command UX overstate current shipped behavior.

For v3.1, the highest-value work is not inventing new architecture. It is aligning the actual product path with the architecture that already exists in code.

## Detailed Matrix

### M1. Genesis Must Become the Authoritative Product Path

**Spec / Product Claim**

- `genesis` is the adaptive front door.
- Genesis should synthesize discovery, context gathering, interview, domain selection, team design, and forge.
- Genesis should be able to produce a team that is meaningfully different from a simple scan-and-compose pass.

**Current Code Status**

- `runGenesis()` in `src/genesis/index.ts` builds a project brief and domain list.
- The CLI command in `src/cli/commands/genesis.ts` calls `runGenesis()`.
- `runGenesis()` ultimately calls `forgeTeam()` from `src/builder/index.ts` and then attaches `domains` and `project_brief` to the returned manifest.
- `designTeam()` in `src/genesis/team-designer.ts` exists, but it is not the authoritative path that writes the final generated output.

**Evidence**

- `src/genesis/index.ts`
- `src/cli/commands/genesis.ts`
- `src/builder/index.ts`
- `src/genesis/team-designer.ts`

**Gap**

Genesis is currently a wrapper around forge rather than the product's canonical design flow. That makes it difficult to trust Genesis-specific concepts like topology selection, approval, or brief-driven team design, because the final written team is still largely derived from the separate forge pipeline.

**v3.1 Recommendation**

Refactor Genesis into the actual write-path authority:

1. `discover(projectRoot)`
2. `runFullScan(projectRoot)` and optional research/integration passes
3. `buildBrief(...)`
4. `designTeam(...)`
5. present proposal
6. confirm approval
7. `writeTeam(...)`

Keep `forge` as the fast, non-interactive path for existing users who want a direct scan-to-team flow.

**Acceptance Criteria**

- `agentforge genesis` writes the team designed by Genesis rather than rerunning `forgeTeam()`.
- Manifest contents differ when interview answers or brief inputs differ, even if the file scan is the same.
- Genesis and Forge remain separate, intentional entrypoints with different behaviors.

**Priority**

`P0`

### M2. Interview Mode and Approval Need Real UX

**Spec / Product Claim**

- Genesis supports adaptive interview mode.
- Genesis presents the proposed team and waits for approval before writing.

**Current Code Status**

- Interview questions are defined in `src/genesis/interviewer.ts`.
- The `--interview` flag exists in `src/cli/commands/genesis.ts`.
- There is no interactive interview runner in the CLI path.
- There is no approval gate before writing `.agentforge/`.

**Evidence**

- `src/genesis/interviewer.ts`
- `src/cli/commands/genesis.ts`
- `commands/genesis.md`

**Gap**

The product promise is interactive and approval-driven, but the actual command is effectively non-interactive and write-through.

**v3.1 Recommendation**

Implement a Genesis interaction layer:

- Ask full interview questions when discovery finds an empty or low-context project.
- Ask targeted gap questions when scan data exists.
- Print a proposed team summary and require confirmation before file output.

This can be implemented first with a simple terminal prompt flow.

**Acceptance Criteria**

- `agentforge genesis --interview` asks questions in the terminal.
- `agentforge genesis` shows the proposed team before writing.
- Users can cancel without modifying `.agentforge/`.

**Priority**

`P0`

### M3. The Universal Brief Must Actually Be Universal

**Spec / Product Claim**

- Universal Forge supports codebase mode, documents mode, hybrid mode, and research/integration context.
- The Project Brief is the universal input to team design.

**Current Code Status**

- `buildBrief()` in `src/genesis/brief-builder.ts` populates:
  - `context.codebase`
  - `context.research`
  - `context.integrations`
- It does **not** currently populate `context.documents`.
- `runFullScan()` in `src/scanner/index.ts` captures `documents`, `integrations`, and `comments`, but not `research` by default.
- `researchProject()` in `src/scanner/web-researcher.ts` is an explicit stub.
- `analyzeDocuments()` in `src/scanner/document-analyzer.ts` only reads `.md` and `.txt`.

**Evidence**

- `src/genesis/brief-builder.ts`
- `src/scanner/index.ts`
- `src/scanner/document-analyzer.ts`
- `src/scanner/web-researcher.ts`

**Gap**

The brief is still software-first. Business/document-heavy projects are only partially represented, and autonomous research remains a documented capability rather than a shipped one.

**v3.1 Recommendation**

Minimum v3.1 scope:

1. Add `documents` into the built brief.
2. Pass scan-derived integrations into the Genesis brief by default.
3. Expand document support beyond `.md` and `.txt`.
4. Either:
   - ship a real research path, or
   - clearly de-scope it in product docs until it is ready.

**Acceptance Criteria**

- A docs-only project produces a populated `project_brief.context.documents`.
- Hybrid projects preserve both codebase and document context.
- `genesis` can meaningfully distinguish document-heavy business work from code-heavy software work.

**Priority**

`P0`

### M4. Collaboration and Topology Must Be Persisted as Product Artifacts

**Spec / Product Claim**

- Collaboration templates, topology selection, and cross-domain bridges are core product behavior.
- Teams should have visible, versioned collaboration structure.

**Current Code Status**

- `selectTopology()` exists in `src/collaboration/topology-selector.ts`.
- `buildBridges()` exists in `src/collaboration/bridge-builder.ts`.
- Collaboration template loading exists in `src/collaboration/template-loader.ts`.
- The main forge pipeline in `src/builder/index.ts` does not load or persist collaboration templates.
- `writeTeam()` in `src/builder/team-writer.ts` does not write a collaboration artifact beyond the delegation graph.

**Evidence**

- `src/collaboration/topology-selector.ts`
- `src/collaboration/bridge-builder.ts`
- `src/collaboration/template-loader.ts`
- `src/builder/team-writer.ts`

**Gap**

Topology exists as logic, not as a first-class generated artifact users can inspect, diff, or tune.

**v3.1 Recommendation**

Make collaboration explicit in generated output:

- add `manifest.collaboration`
- persist selected template metadata
- persist merged topology details in `config/delegation.yaml`
- distinguish structural graph intent from agent-template `can_delegate_to` edges

**Acceptance Criteria**

- Generated `team.yaml` includes collaboration metadata.
- Users can identify which template was selected and why.
- Cross-domain bridges are visible in generated output.

**Priority**

`P1`

### M5. The Default Runtime Path Must Use the v3 Session Runtime

**Spec / Product Claim**

- Runtime execution is cost-aware, self-improving, and session-driven.
- Feedback, decisions, and session memory should be first-class runtime outputs.

**Current Code Status**

- `invoke` and `delegate` in `src/cli/commands/` construct the legacy `Orchestrator`.
- The legacy `Orchestrator` in `src/orchestrator/index.ts` is a useful facade, but not the richest runtime implementation.
- The richer runtime is `OrchestratorV3` in `src/orchestrator/orchestrator-v3.ts`.
- The highest-level runtime composition is `AgentForgeSession` in `src/orchestrator/session.ts`.

**Evidence**

- `src/cli/commands/invoke.ts`
- `src/cli/commands/delegate.ts`
- `src/orchestrator/index.ts`
- `src/orchestrator/orchestrator-v3.ts`
- `src/orchestrator/session.ts`

**Gap**

The product does not currently exercise the most advanced runtime path in normal use, so the most important v3 capabilities exist beside the user experience rather than inside it.

**v3.1 Recommendation**

Make the session runtime the default path for `invoke` and `delegate`, with a temporary legacy fallback flag for safety.

**Acceptance Criteria**

- Normal CLI execution creates session-side decisions, knowledge entries, and runtime cost data.
- Session-backed execution is the default documented behavior.
- Legacy runtime remains available only as a migration fallback.

**Priority**

`P0`

### M6. Multi-Agent Runtime Needs a Real Control Loop

**Spec / Product Claim**

- AgentForge should support true multi-agent collaboration with routing, handoffs, review, and bounded retry/escalation.

**Current Code Status**

- The repository has many orchestration primitives:
  - progress ledger
  - loop guard
  - event bus
  - handoff manager
  - speaker selector
  - review enforcer
- These are composed into facades, but there is no top-level controller loop that keeps executing until the task is satisfied or bounded out.

**Evidence**

- `src/orchestrator/index.ts`
- `src/orchestrator/progress-ledger.ts`
- `src/orchestrator/loop-guard.ts`
- `src/orchestrator/speaker-selector.ts`
- `src/orchestrator/review-enforcer.ts`

**Gap**

The runtime is currently a toolkit of managers rather than a fully operational autonomous controller.

**v3.1 Recommendation**

Add an explicit execution loop with:

1. task start / ledger initialization
2. next-speaker selection
3. action execution
4. handoff/review/event handling
5. health check
6. bounded replanning or escalation
7. terminal success/failure state

**Acceptance Criteria**

- One command can drive a bounded multi-step workflow without manual orchestration by the caller.
- Loop limits are enforced in actual runtime control flow.
- Review and handoff paths influence execution rather than just existing as primitives.

**Priority**

`P1`

### M7. Ship One Real Integration Path

**Spec / Product Claim**

- External tool integration is native, not bolted on.

**Current Code Status**

- `generateMcpConfig()` in `src/integrations/mcp-config-generator.ts` is real.
- `IntegrationLayer.dispatch()` in `src/integrations/integration-layer.ts` is explicitly stubbed.

**Evidence**

- `src/integrations/mcp-config-generator.ts`
- `src/integrations/integration-layer.ts`

**Gap**

The platform can detect integrations and generate config, but not execute real actions through them.

**v3.1 Recommendation**

Choose one provider and ship it end-to-end. Recommended order:

1. GitHub
2. Jira
3. Confluence
4. Slack

GitHub is the cleanest first slice because issue/comment workflows are easy to validate and the repository already has relevant product semantics.

**Acceptance Criteria**

- A supported integration can be detected, configured, and invoked from runtime.
- One end-to-end integration workflow is covered by tests.
- Integration docs are narrowed to what is actually operational.

**Priority**

`P1`

### M8. Cost Reporting Must Reflect Actual Runtime Behavior

**Spec / Product Claim**

- Cost awareness is central to v3.
- The product should show real token usage, tier distribution, and routing efficiency.

**Current Code Status**

- `runCostAware()` in `src/orchestrator/cost-aware-runner.ts` does real cost estimation, routing, and recording.
- `OrchestratorV3.runAgent()` currently always goes through `runCostAware()` even when `enableCostAwareRouting` is false.
- `cost-report` in `src/cli/commands/cost-report.ts` looks for `.agentforge/cost-data.json`.
- No normal runtime path currently writes that file.

**Evidence**

- `src/orchestrator/cost-aware-runner.ts`
- `src/orchestrator/orchestrator-v3.ts`
- `src/cli/commands/cost-report.ts`
- `commands/cost-report.md`

**Gap**

Execution logic has outpaced persistence and reporting. One config flag is misleading, and the user-facing report does not match actual runtime telemetry.

**v3.1 Recommendation**

1. Make `enableCostAwareRouting` truly branch behavior.
2. Persist session/runtime cost artifacts to disk.
3. Make `cost-report` consume the real stored artifacts.
4. Add projected team-cost output when no runtime data exists yet.

**Acceptance Criteria**

- After live runs, `cost-report` shows actual by-tier and by-agent usage.
- With no runtime data, `cost-report` shows a projected cost profile based on team routing.
- The command behavior matches `commands/cost-report.md`.

**Priority**

`P1`

### M9. Clarify the Two Reforge Systems

**Spec / Product Claim**

- Reforge is the continuous improvement loop.

**Current Code Status**

- `src/reforge/index.ts` implements scan-driven team regeneration and diffing.
- `src/reforge/reforge-engine.ts` implements runtime agent overrides and structural proposals from feedback.

**Evidence**

- `src/reforge/index.ts`
- `src/reforge/reforge-engine.ts`

**Gap**

Two useful but different products share the same noun:

- team regeneration from project deltas
- runtime agent tuning from feedback

This is conceptually powerful, but user-facing semantics are blurry.

**v3.1 Recommendation**

Choose one of two approaches:

- rename the concepts, or
- keep `reforge` as the umbrella term but make the submodes explicit in CLI/docs

Recommended product language:

- `team reforge` or `reforge team`
- `agent tune` or `reforge agent`

**Acceptance Criteria**

- CLI help, docs, and generated artifacts make it obvious which reforge path happened.
- Devs can explain the difference without referencing source files.

**Priority**

`P1`

### M10. Align Docs, Command Surface, Packaging, and Metrics With Reality

**Spec / Product Claim**

- Public documentation should describe the actual product.
- Plugin metadata and CLI command names should match shipped UX.
- Benchmarks and counts should be trustworthy.

**Current Code Status**

- `plugin.json` advertises `ask`, but the actual command is `invoke`.
- `docs/design.md` still mentions commands like `init` and `export` that do not exist.
- `README.md` and other docs contain stale inventory counts.
- `package.json` has no `bin` field for normal CLI packaging.
- There are strong unit tests, but no CLI end-to-end suite and no plugin packaging/install validation.

**Evidence**

- `plugin.json`
- `src/cli/index.ts`
- `src/cli/commands/invoke.ts`
- `docs/design.md`
- `README.md`
- `src/benchmark/v3-launch-gate.ts`
- `package.json`

**Gap**

Documentation and packaging maturity lag behind code maturity, which creates avoidable trust erosion.

**v3.1 Recommendation**

1. Add `ask` as an alias for `invoke`, or rename `invoke` to `ask`.
2. Add `bin` to `package.json`.
3. Remove or implement stale documented commands.
4. Replace hardcoded counts with script-generated metrics.
5. Add CLI E2E tests and at least one packaging/install validation path.

**Acceptance Criteria**

- `--help`, plugin metadata, README, and command docs all match.
- Package can be installed/executed as a normal CLI.
- Public metrics are derived, not manually maintained.

**Priority**

`P0`

## Final Assessment

AgentForge does not need a new architectural direction for v3.1. It needs convergence.

The codebase already contains the core ingredients of the product described in the specs:

- domain-aware team generation
- Genesis concepts
- collaboration primitives
- cost-aware routing
- feedback loops
- reforge logic
- runtime session infrastructure

The main v3.1 challenge is to:

1. make the intended paths authoritative,
2. wire the advanced runtime into the default product surface,
3. turn existing scaffolds into first-class shipped behavior, and
4. bring docs and command UX back into alignment with the actual system.

If those four things are done well, v3.1 will feel like a coherent platform rather than a strong codebase with partially productized subsystems.
