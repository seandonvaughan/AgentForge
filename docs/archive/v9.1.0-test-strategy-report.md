## Executive Summary

Sprint v9.1.0 delivered comprehensive memory-wiring infrastructure (11 items), dashboard completion (7 items), with **zero reported failures**. However, critical API endpoints and phase handler logic added in this sprint **lack direct unit/integration test coverage**, creating a gap between implementation completion and test verification.

**Key Risk**: Three new approval-related API endpoints in `/cycles.ts` are untested, and the gate phase memory entry logic in `phase-handlers.ts` has limited edge-case coverage.

## Risk Assessment by Changed File

### 🔴 HIGH RISK

#### `packages/server/src/routes/v5/cycles.ts` (28 lines changed)
**What Changed**: 
- Added `hasApprovalPending` flag to GET `/api/v5/cycle-sessions`
- Added GET `/api/v5/cycles/:id/approval` (fetch pending approval data)
- Added POST `/api/v5/cycles/:id/approve` (submit approval decision)

**Risk Assessment**:
- **Gap**: Zero tests for these three endpoints. Existing `cycles-detail.test.ts` covers only GET `/api/v5/cycles/:id` detail view.
- **Impact**: Approval flow is critical to the autonomous loop's human-in-the-loop safeguard. Silent failures here could allow unapproved changes to ship.
- **Edge Cases Not Covered**:
  - Concurrent POST /approve requests (race condition safety)
  - Malformed request body (missing/invalid approvedItemIds)
  - POST /approve when already decided (409 conflict) — code handles it, but no test
  - Missing approval-pending.json (404 handling) — no test
  - Workspace isolation: `hasApprovalPending` flag across different workspaceIds

**Recommended Tests**:
- `test-cycles-approval-endpoints-happy-path.ts` (3 tests)
  - GET /api/v5/cycles/:id/approval returns pending approval or 404
  - POST /api/v5/cycles/:id/approve with approveAll flag writes approval-decision.json
  - POST /api/v5/cycles/:id/approve with itemId array writes partial approval
- `test-cycles-sessions-approval-flag.ts` (4 tests)
  - GET /api/v5/cycle-sessions includes hasApprovalPending=true when pending exists
  - hasApprovalPending=false when no pending or already decided
  - Workspace-specific approval state isolation

#### `packages/server/src/lib/phase-handlers.ts` (77 lines changed)
**What Changed**:
- Added memory entry writing to `runGatePhase()` with structured gate-verdict payload
- New helper `extractFindingLines()` parses review text for CRITICAL/MAJOR findings
- Gate verdict captured before LLM re-run to prevent decision overwrites

**Risk Assessment**:
- **Gap**: Integration test exists (`gate-phase-memory.test.ts`, 9 tests) but **lacks edge case coverage**.
- **Impact**: Gate phase is the permission gate for the entire autonomous loop. Memory logic intertwines verdict extraction with review text parsing — a parsing bug here silently loses critical findings.
- **Edge Cases Not Covered**:
  - Empty/null review text (review phase never ran, or failed)
  - Review text with NO CRITICAL/MAJOR findings (empty arrays passed to memory)
  - Malformed JSON in sprint file (readSprint() returns null or corrupted)
  - Memory write failure in `writeMemoryEntry()` (silently caught, but no test of fallback)
  - Very large review text (>500k characters) — slicing to 500 chars in rationale, but no test of overflow
  - Unicode/binary in review text — extractFindingLines splits by `\n`

**Recommended Tests**:
- `test-gate-phase-finding-extraction-edge-cases.ts` (7 tests)
  - extractFindingLines with no matching severity strings (empty result)
  - extractFindingLines with >10 matches (cap at 10 is enforced)
  - extractFindingLines with mixed CRITICAL and MAJOR (both extracted separately)
  - Review text with unicode/emoji in finding lines
  - Gate phase runs when pre-gate review is missing
  - Gate phase memory write failure doesn't abort phase (caught in try-catch)
  - Very large review text (>50KB) slices finding lines without OOM

### 🟡 MEDIUM RISK

#### `packages/server/src/routes/v5/dashboard-stubs.ts` (281 lines added)
**What Changed**:
- Expanded GET `/api/v5/memory` with query filters: `?type=`, `?since=`, `?agentId=`
- Workspace-scoped memory cache
- Per-workspace `/api/v5/flywheel` cache
- New session metrics (success rate, confidence trend) in flywheel computation

**Risk Assessment**:
- **Gap**: Memory endpoint has tests (`memory.test.ts`, 19 tests) but **new filtering logic untested**.
- **Impact**: Memory dashboard may show wrong results if filters fail. Workspace caching could cause cache collisions if workspaceId param is missing/malformed.
- **Edge Cases Not Covered**:
  - `?since=` with invalid ISO-8601 date (code ignores via `isNaN()`, but no test of behavior)
  - `?since=` with date far in future (no entries match, empty result)
  - `?type=` with non-existent type (empty result expected)
  - Multiple filters combined (type + since + agentId) — interaction testing
  - Very large memory directory (>10K files) — does cache cap at MEMORY_LIMIT work?
  - Missing `.agentforge/memory/` directory (code handles existsSync, but no test)
  - Concurrent reads during background JSONL writes (race condition on read)

**Recommended Tests**:
- Add to existing `memory.test.ts`:
  - Test each filter individually (type, since, agentId)
  - Test combined filters (e.g. type=cycle-outcome AND since=2026-04-09T00:00:00Z)
  - Test invalid since date format (should be ignored)
  - Test empty result sets (no entries match filter)

#### `src/autonomous/review-phase-handler.ts` (46 lines changed)
**What Changed**:
- Added `line`, `summary` fields to ReviewFinding (replacing `message`)
- Added `fixSuggestion` optional field
- Structured `metadata` payload in memory entry with {file, line, severity, summary, fixSuggestion}

**Risk Assessment**:
- **Gap**: Tests exist (`review-phase-handler.test.ts`, 25 tests) and cover the new schema, but **metadata serialization untested**.
- **Impact**: Execute phase injector relies on metadata to extract findings. If metadata is null/undefined/malformed, injector can't surface specific code locations.
- **Edge Cases Not Covered**:
  - line=0 (valid line number, but falsy in JS — code uses `line != null` correctly, but test should verify)
  - fixSuggestion with special characters (JSON serialization edge case)
  - Very long summary/fixSuggestion strings (>10KB) — memory entry size explosion
  - Circular references in metadata (shouldn't happen, but JSON.stringify would silently fail)

**Recommended Tests** (add to existing suite):
- Test metadata field is present in memory entry JSON
- Test metadata with null/undefined fixSuggestion serializes correctly
- Test line=0 is not treated as missing (falsy but valid)
