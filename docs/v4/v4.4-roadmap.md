---

## Architectural Principle

AgentForge is a **Claude Code plugin**. It runs inside Claude Code and inherits Claude
Code's existing session, authentication, and model access (including Max Pro). There is
no separate API key — agents execute by being dispatched as Claude Code subagents via the
`Agent` tool, using the agent's YAML as the subagent prompt and model routing to pick
opus/sonnet/haiku. This is the correct execution model.

The future product that calls the Anthropic API directly is a separate project.

## P0 — Real Agent Execution via Claude Code (must ship)

### P0-1: `/agentforge:invoke` dispatches a real Claude Code subagent
**Owner:** architect + coder
The `invoke` command currently stubs responses. Make it dispatch a real subagent using
Claude Code's `Agent` tool pattern — the same mechanism Claude Code uses internally.

The command file (`commands/invoke.md`) should produce a prompt that:
- Loads the target agent's YAML (`name`, `system_prompt`, `model`, `skills`)
- Sets the subagent's system context from the agent's `system_prompt`
- Routes to the correct model via the `model` field (opus/sonnet/haiku → claude-opus-4-6 / claude-sonnet-4-6 / claude-haiku-4-5)
- Passes the user's task as the subagent prompt
- Records the result in `.agentforge/sessions/<id>.json`

No `@anthropic-ai/sdk` needed — Claude Code handles the call.

### P0-2: Session persistence to disk
**Owner:** coder
`V4Session` must write its state to `.agentforge/sessions/` on every state change so the
dashboard and flywheel can read real outcomes.
- `session-<id>.json` — agent, task, response, model, timestamp, token estimates
- `sessions/index.json` — summary list read by dashboard Session Timeline
- Hook into the post-invoke step to write before the command returns

### P0-3: Bus events from real invocations
**Owner:** coder + meta-architect
Every real `invoke` run emits lifecycle events to `V4MessageBus`, which writes them to
`.agentforge/data/bus-events.json` via a new `BusFileAdapter`.
- Topics: `agent.invoked`, `agent.responded`, `agent.error`
- Dashboard Bus Monitor loads from `bus-events.json` (replaces simulation for real runs)
- `BusFileAdapter`: append-only JSON lines file, max 500 entries (rolling)

### P0-4: Fix forge hook team.yaml overwrite
**Owner:** devops-engineer
The `/agentforge:forge` command resets `team.yaml` to the 17 scanned agents, losing
all 37 manual additions. Fix the forge command to merge rather than replace:
- Read existing `team.yaml` before writing
- Preserve existing agents not found in the scan (manual additions)
- Preserve `team_size`, `version`, `delegation_graph` custom entries
- Only add newly discovered agents, never remove existing ones

## P1 — Intelligence Improvements

### P1-1: Enhanced TF-IDF → semantic search with synonym expansion
**Owner:** rd-lead + coder
Rather than embedding API calls (separate product concern), push `SemanticSearch` further:
- Expand the synonym map with domain-specific AgentForge terms
- Add bigram/trigram phrase matching for compound concepts
- BM25 ranking to replace raw TF-IDF scores
- Benchmark: >98% recall on 50 AgentForge-specific queries

### P1-2: MetaLearningEngine real feedback loop
**Owner:** meta-architect + rd-lead
`MetaLearningEngine` classifies patterns but doesn't act on them. Connect it to real
session outcomes:
- Read completed sessions from `.agentforge/sessions/index.json`
- Auto-promote agents meeting criteria (tier 2→3: 5+ tasks, zero regressions)
- Write promotion events to `.agentforge/data/autonomy-tiers.json` (feeds dashboard)
- Emit `flywheel.agent.promoted` bus event → triggers dashboard notification

### P1-3: Multi-agent fan-out via Claude Code parallel subagents
**Owner:** architect + coder
The `/agentforge:invoke` command gains a `--parallel` mode that dispatches the same task
to multiple agents simultaneously using Claude Code's parallel agent capability.
- `--parallel cto,architect,meta-architect` dispatches all three
- Each runs with its own system prompt and model
- Results returned side-by-side for comparison
- `team-reviewer` agent auto-summarizes differences

### P1-4: Knowledge ingestion from codebase
**Owner:** rd-lead + researcher
Auto-index the project's TypeScript exports into the memory system.
- Walk `src/**/*.ts`, extract exported names + JSDoc comments
- Store in `.agentforge/knowledge/codebase-index.json`
- Agents query `SemanticSearch` against this index before acting on code tasks
- Dashboard Memory Browser shows both memory entries and knowledge index
