# AgentForge v4.5 Roadmap

**Status:** Active
**Sprint:** Autonomous v4.5
**Budget:** $600
**Team:** 37 agents (6 Opus, 27 Sonnet, 4 Haiku)
**Theme:** *Agents that think together, learn from everything, and never forget.*

---

## Focus

v4.5 transforms AgentForge from a collection of capable individual agents into a genuinely collaborative system. Where v4.4 gave agents the ability to execute, v4.5 gives them the ability to *coordinate*: real task pipelines between agents, persistent cross-session memory that survives Claude Code restarts, a workflow engine that chains multi-agent sequences, and an agent-to-agent delegation protocol that lets a CTO agent actually hand work to a coder agent and get results back.

The founder should see a system where typing `/agentforge:invoke cto "build feature X"` triggers a visible chain of delegation, execution, and synthesis -- not a single agent in isolation.

---

## Architectural Principle

Same as v4.4: AgentForge is a Claude Code plugin. All agent execution happens via Claude Code subagent dispatch. No external API calls. No separate process. Zero new npm dependencies (Iron Law 5).

## P0 -- Must Ship (6 items)

### P0-1: Task Pipeline Engine
**Owner:** architect + coder
**Why:** Right now, agents are invoked one at a time. Real work requires chains: CTO decomposes a task, assigns subtasks to coder and researcher, collects results, synthesizes. The pipeline engine makes this possible.

Implementation:
- New `src/pipeline/task-pipeline.ts`:
  - `TaskPipeline` class with stages: each stage has an `agentId`, `task` template, `dependsOn` list
  - `PipelineExecutor` walks the DAG, executing stages in topological order
  - Parallel stages (no dependency) run concurrently via `Promise.allSettled`
  - Each stage result is passed as context to dependent stages
  - Pipeline status: `pending | running | completed | failed | partial`
  - Emits bus events: `pipeline.started`, `pipeline.stage.completed`, `pipeline.completed`
- New `src/types/pipeline.ts` with full type definitions
- Pipeline results persisted to `.agentforge/pipelines/<pipelineId>.json`

### P0-2: Agent-to-Agent Delegation Protocol
**Owner:** cto + architect
**Why:** The `delegation_graph` in `team.yaml` defines who can delegate to whom, but there is no runtime protocol. This makes it real.

Implementation:
- New `src/delegation/delegation-protocol.ts`:
  - `DelegationRequest`: `{ from, to, task, context, constraints, budget }`
  - `DelegationResult`: `{ success, response, tokensUsed, durationMs }`
  - `DelegationProtocol` class validates delegation authority against `team.yaml` graph
  - Checks: agent exists, delegation path is authorized, budget is within envelope
  - Records every delegation in `.agentforge/data/delegation-log.json`
  - Emits `delegation.requested`, `delegation.completed`, `delegation.rejected` bus events
- Integrated into pipeline engine so pipeline stages use delegation protocol

### P0-3: Cross-Session Memory Persistence
**Owner:** rd-lead + coder
**Why:** Every time Claude Code restarts, agents lose all context. This wires the KnowledgeStore and MetaLearningEngine to persist and reload automatically.

Implementation:
- New `src/memory/session-memory-manager.ts`:
  - On session end: serialize key learnings, decisions, patterns to `.agentforge/memory/session-history.json`
  - On session start: load previous session history, inject as context to agents
  - Rolling window: keep last 20 sessions, compact older ones into summaries
  - Each session summary includes: what agents did, what they learned, what failed
  - Categories: `task-outcome`, `pattern-discovered`, `agent-interaction`, `error-recovery`
- Wire into `AgentForgeSession.create()` to auto-load, and `AgentForgeSession.end()` to auto-save

### P0-4: Workflow Templates (Predefined Multi-Agent Sequences)
**Owner:** skill-designer + coder
**Why:** Common multi-agent patterns should be one-click. "Code review workflow", "Feature implementation workflow", "Bug investigation workflow" -- predefined pipeline templates.

Implementation:
- New `src/pipeline/workflow-templates.ts`:
  - `WorkflowTemplate` interface: name, description, stages[], requiredAgents[]
  - Built-in templates:
    - `code-review`: researcher (gather context) -> coder (implement) -> team-reviewer (review) -> coder (address feedback)
    - `bug-investigation`: debugger (reproduce) -> researcher (root cause) -> coder (fix) -> qa (verify)
    - `feature-design`: architect (design) -> cto (approve) -> coder (implement) -> team-reviewer (review)
    - `knowledge-sync`: researcher (scan codebase) -> meta-architect (extract patterns) -> ceo (strategic summary)
  - `WorkflowRegistry` class for listing and instantiating templates
  - Templates are parameterizable: `workflow.instantiate("code-review", { targetFile: "src/foo.ts" })`

### P0-5: BusFileAdapter Real-Time Event Streaming
**Owner:** coder + devops-engineer
**Why:** v4.4 created `BusFileAdapter` but it is not integrated with the V4MessageBus. This wires them together so every bus event is persisted to disk, making the dashboard genuinely live.

Implementation:
- New `src/communication/bus-file-adapter.ts`:
  - `BusFileAdapter` class wraps `V4MessageBus`
  - On every publish: append event to `.agentforge/data/bus-events.jsonl` (JSON lines format)
  - Rolling window: max 1000 entries, oldest entries pruned on write
  - `loadEvents(since?: string)`: read events since a timestamp for dashboard consumption
  - Global listener on the bus captures all messages without subscribing to individual topics
- Wire into session creation so every v4 bus event is automatically persisted

### P0-6: Agent Health Heartbeat System
**Owner:** devops-engineer + coder
**Why:** The V4HealthCheck exists but has no probes registered. The AgentStatusFile schema defines heartbeats but nothing writes them. This makes agent health observable.

Implementation:
- New `src/observability/agent-heartbeat.ts`:
  - `AgentHeartbeat` class: writes status files to `.agentforge/status/<agentId>.json`
  - Records: state, activeTask, lastHeartbeat, capabilities, model tier
  - `StalenessDetector` integration: marks agents offline if heartbeat > 2 minutes stale
  - `V4HealthCheck` probes auto-registered for: bus health, session health, memory health, pipeline health
  - `getTeamHealth()`: aggregate health across all agents with summary metrics
- Persisted health snapshots to `.agentforge/data/health-snapshots.json` for dashboard trend lines

## P1 -- High Value (5 items)

### P1-1: Smart Task Decomposition
**Owner:** architect + rd-lead
**Why:** The current `decomposeTask` splits on "and" / commas, which is crude. Real decomposition should use the task structure and agent capabilities to plan subtask assignments.

- Enhance `SmartRouter` to analyze task complexity and recommend decomposition
- Match subtask requirements to agent capabilities from the team manifest
- Return a decomposition plan: `[{ subtask, recommendedAgent, rationale }]`
- Integrate with pipeline engine for automatic fan-out

### P1-2: MetaLearningEngine Pattern Persistence
**Owner:** meta-architect + rd-lead
**Why:** MetaLearningEngine extracts patterns and generates insights but only in-memory. Patterns should persist across sessions.

- Persist pattern stats to `.agentforge/flywheel/patterns.json`
- Persist knowledge graph to `.agentforge/flywheel/knowledge-graph.json`
- Load on engine initialization
- Generate weekly (session-count based) insight reports to `.agentforge/flywheel/insights/`

### P1-3: `/agentforge:workflow` Command
**Owner:** coder + skill-designer
**Why:** Users need a way to invoke workflow templates from the CLI.

- New `commands/workflow.md` slash command
- Usage: `/agentforge:workflow code-review --target src/foo.ts`
- Lists available workflows when called with no arguments
- Shows workflow progress and stage completions in real-time

### P1-4: Delegation Authority Visualization
**Owner:** dashboard-architect (dashboard implementation deferred)
**Why:** The delegation graph is invisible. Users should see who can delegate to whom.

- Data preparation: export delegation graph as a structured JSON for dashboard consumption
- Write to `.agentforge/data/delegation-graph.json` with enriched metadata (success rates, task counts)
- Dashboard section: deferred to dashboard agent

### P1-5: Error Recovery and Retry Logic
**Owner:** debugger + coder
**Why:** When an agent invocation fails, the system gives up. Real systems retry with backoff.

- `RetryPolicy` class: configurable max retries, backoff strategy (linear/exponential)
- Integration with pipeline executor: failed stages retry before marking pipeline as failed
- Circuit breaker: after N consecutive failures for an agent, skip and report
- Error classification: transient (retry) vs permanent (skip) vs budget-exceeded (halt)
